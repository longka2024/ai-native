# Longka 内容资产工程 v1 设计

## 目标

v1 先做 X 单平台的内容资产工程闭环：

```text
X 对标账号
-> 真实采集帖子
-> 原始样本入库
-> 去重和来源追溯
-> 抽取内容单元
-> 生成主题地图
-> 生成今日内容雷达
```

阶段性成品是“今日内容雷达页”，让团队看到可用结果，而不是只看到数据库和采集日志。

## 不做

- 不做小红书评论采集。
- 不做 RSS、Reddit、Podcast、Newsletter。
- 不做定时监控。
- 不做文案、图片、视频生成。
- 不做模型训练。

## 核心对象

- `content_sources`：信息源，v1 只接 X 账号。
- `raw_content_samples`：原始样本，所有真实采集结果都进入。
- `content_units`：内容单元，类型只允许 `QST / CON / OPI / CAS / SOL`。
- `content_relations`：显式关系，只允许“回应 / 解释 / 证明 / 冲突”。
- `topic_maps`：主题地图。
- `assembly_drafts`：选题装配稿。
- `processing_ledger`：处理状态。

## v1 页面

左侧新增独立页面：`内容资产工程`。

页面包含：

1. 今日内容雷达。
2. 对标账号库。
3. 原始样本库。
4. 内容单元库。
5. 主题地图和选题装配稿。

## 数据规则

- 采集失败必须显示真实失败原因，不允许假数据顶替。
- 每条样本、内容单元、装配稿都必须能追溯到原帖链接。
- 原创或直接发布内容优先进入内容单元抽取。
- 转发、引用、回复、来源不清的内容默认只留在原始样本库。
- 今日内容雷达必须显示采集数、新增数、过滤数、内容单元数、主题地图数、装配稿数。

## 最小验收

```text
添加 3 个 X 账号
-> 点击立即采集
-> 每个账号真实采集 20 条
-> 原始样本入库
-> 自动分档为 QST/CON/OPI/CAS/SOL
-> 生成至少 1 张主题地图
-> 生成 3 个选题装配稿
-> 页面可打开来源原帖
```
