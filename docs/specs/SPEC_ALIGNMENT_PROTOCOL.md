# Longka Spec 对齐流程

## 1. 目的

这个流程用于把模糊需求变成可开发、可验收、可唤醒的 spec。

它解决的问题是：

- 用户只记得大概方向，不一定能一次说完整标准。
- AI 容易根据大方向自行扩展，导致跑偏。
- 聊天里对齐过的内容，如果不落文件，后续很难唤醒。

以后 Longka 新功能、重构、关键改造，都必须按这个流程走。

## 2. 分工

### 用户负责

- 提出目标、感觉、业务判断、是否满意。
- 对关键选择拍板。
- 确认最终 spec。

用户不需要一次性写完整 PRD。

### AI 负责

- 把模糊需求拆成具体问题。
- 一次只问一个关键问题。
- 给出 2-3 个可选方案和推荐理由。
- 把用户确认过的内容整理成 spec。
- 开发前先读取对应 spec。
- 如果开发中发现需要扩大范围，必须暂停并重新确认。

AI 不允许把“方向对齐”当成“开发 spec”直接开写。

## 3. 五条件合格 Spec

一个 spec 合格，必须包含 5 个条件。

### 3.1 名字

用于后续唤醒。

示例：

- Day1 文案版本管理器
- 三天稳定版计划
- 内容资产模块一期
- 今日工作台 SOP 稳定版

### 3.2 触发词

用户后续记不清完整名字时，可以用触发词唤醒。

示例：

- “按 Day1”
- “继续优化不要只追加建议”
- “不丢稿”
- “三天稳定版”
- “今日工作台别跑偏”

### 3.3 范围

这次要做什么。

必须写成可执行动作，不能只写抽象目标。

示例：

- 生成正文后保存第 1 版。
- 点击继续优化后基于当前正文生成第 2 版。
- 每一版可回看、恢复、确认。

### 3.4 禁止范围

这次不做什么。

禁止范围非常重要，用来防止 AI 顺手扩大改造。

示例：

- 不改采集模块。
- 不改母题筛选。
- 不重构标题库。
- 不进入图文闭环。

### 3.5 验收标准

怎么证明做好了。

必须能通过页面、数据、命令或用户操作验证。

示例：

- 生成第 1 版后版本列表出现。
- 继续优化后新增第 2 版。
- 第 1 版仍能恢复。
- 第 2 版能确认。
- 优化结果不是只在文末追加建议。

## 4. 标准对齐流程

### 第一步：用户说大概目标

用户可以说得很模糊。

示例：

```text
我想把继续优化这块做好，不要老是丢稿。
```

### 第二步：AI 复述理解

AI 必须先复述：

```text
我理解你要解决的是：文案生成后能反复优化，但不能丢失上一版，也不能只追加建议。
```

### 第三步：AI 一次问一个关键问题

不要一次问一堆。

优先问会影响产品形态的问题。

示例：

```text
版本历史是只保存在当前页面，还是要写入数据库，刷新后也能回来？
```

### 第四步：AI 给 2-3 个方案

示例：

```text
A. 先做页面内版本历史，最快。
B. 写入数据库，刷新后也能恢复，稍慢。
C. 同时做数据库和发布记录，范围偏大。

我建议先做 B，因为“不丢稿”最好不能只靠页面内存。
```

### 第五步：用户拍板

用户可以说：

```text
先做 A。
```

或：

```text
听你的，做 B。
```

### 第六步：AI 整理成五条件 spec

AI 必须整理：

```text
名字：
触发词：
范围：
禁止范围：
验收标准：
```

### 第七步：用户确认

只有用户明确确认后，spec 才生效。

确认语可以是：

- “确认”
- “可以”
- “就按这个”
- “A”
- “听你的”

### 第八步：AI 写入文件

写入：

```text
docs/specs/YYYY-MM-DD-<spec-name>.md
```

同时更新：

```text
docs/specs/SPEC_INDEX.md
```

### 第九步：进入开发

开发前 AI 必须复述：

```text
我已读取 <spec name>。
本次只做...
不做...
验收标准是...
```

## 5. 如果用户记不清 spec

用户可以只说大概：

```text
之前我们是不是对齐过那个不丢稿的？
```

AI 必须：

1. 搜索 `docs/specs`。
2. 列出最可能相关的 spec。
3. 说明推荐使用哪一个。
4. 等用户确认后再开发。

## 6. 如果聊天中已经有方向对齐

如果用户认可过一段规划，但没有正式 spec，处理方式是：

1. AI 不直接开写。
2. AI 把那段规划整理成五条件 spec。
3. 用户确认。
4. 写入文件。
5. 再开发。

方向对齐不是开发 spec。

方向对齐只有写成五条件 spec 后，才是可执行标准。

## 7. Longka 约定

以后 Longka 项目里，“对齐 spec”默认指：

```text
逐步追问
-> 用户判断
-> AI 整理五条件
-> 用户确认
-> 写入 docs/specs
-> 开发前读取
```

如果 AI 跳过这个流程，用户可以直接说：

```text
你没有按 Spec 对齐流程。
```

AI 必须停止当前扩展，回到本流程。
