# SPEC：采集架构商用定调（采集内部化 + 工具分工）

- **名字**：采集架构商用定调
- **触发词**："采集架构定调" / "客户不碰cookie" / "江湖补充档" / "采集分工" / "黄金组合"
- **日期**：2026-06-14
- **状态**：已拍板（2026-06-14）
- **性质**：架构决策 / 对前两期采集 spec 的修正补充（不推翻已完成的管线，只修正方向与工具定位）
- **上游**：`2026-06-12-collection-pipeline-rebuild-spec.md`（一期）、`2026-06-12-signal-driven-collection-spec.md`（二期）

---

## 0. 一句话目标

把"商用多租户下采集到底谁来做、用什么工具、怎么不封号"一次定死：采集是平台内部供料，客户只消费内容工厂；高危搜索交给健壮工具，自有账号只做低危深挖。

---

## 1. 背景（为什么要这份定调）

- 产品面向**中小商户、不懂技术**：他们看到的是"能产文案 / 小红书图文 / 视频"的内容工厂，不该接触 cookie / 扩展 / F12。
- 中途换的模型把方向带偏：方案 D 让"工作台手动贴 cookie → xlsx"，反而从已部署的 CookieBridge 倒退，且把采集做成了面向客户的功能。
- 实测：自己开 CDP/Pro 浏览器搜关键词，几次就弹图片验证（CDP/playwright 被检测 + 搜索是最高危接口）；而江湖工具箱（本地 win 软件、扫码登录）健壮、可用很久。

---

## 2. 已锁决策（范围）

### 2.1 采集彻底内部化
- 用**平台自有的一小撮采集小号**集中采集，喂共享样本库 `longka_content_samples`，服务所有客户。
- **客户全程不碰 cookie / 扩展 / F12**，只输入行业/选题、消费成品。
- 采集与客户数解耦：1000 客户共用一条管线，服务器负载恒定，不随人数爆炸。
- 方案 D 的"工作台账号管理 UI + `crawler_cookies_account` 表"**降级为平台内部运营工具**（管你自己那几个采集号），不是客户功能。

### 2.2 工具分工（写死，避免反复）

| 角色 | 工具 | 边界 |
|---|---|---|
| 发现（无登录/零封号，重度用） | TrendRadar（4线）+ AI HOT（仅 AI 线） | 只给标题+链接，不深抓 |
| 高危搜索取爆款（人工） | 江湖工具箱 | 本地 win 软件，扫码登录，健壮；导出 Excel 人工挑 |
| 低危深挖（保号核心） | MediaCrawlerPro | **退出关键词搜索**，只做"详情+评论+创作者主页"；少而准 + 小号轮换 |
| 网页正文（免费） | 内置 fetch + defuddle → 回退 XCrawl | 信号点进去抓单篇资讯正文 |
| 临时调研（不入库） | Firecrawl | 仅协助查资料，不进生产管线 |

- **海外工具（XCrawl 付费 / AI HOT / last30days）只服务 AI 自媒体 1 条线**；另 3 条线（美容 / 私校留学 / 女性成长）纯国内，海外工具零价值。

### 2.3 江湖工具箱：退役 → 手动补充档
- 一期 spec §2 把江湖列为"退役"，**本 spec 修正为"手动补充档"**。
- 链路：江湖导出 Excel → `POST /api/collectors/local-platform/import-batch` → 写统一表 `longka_content_samples`（`collectorType=local_platform_helper`，**带 workspace**）→ hot30 / 选题 → 二创。
- 人工、按需、不进自动闭环——对二创"人工挑爆款"反而更合适，且**不烧自有 Pro 号**。

### 2.4 信号 → 正文两条路（别记混）
- TrendRadar 热榜（资讯/新闻链接）→ `/api/fetch-article` 抓**网页正文**（内置 fetch 免费，回退 XCrawl，回退 hot30）。**已做好。**
- 社交爆款帖正文 + 评论 → 江湖 / MediaCrawlerPro，**另一条手动路**。

### 2.5 封号规避原则
- 采集号用**专门小号**（可丢弃），不用本人重要账号；备冷号轮换。
- 降频 + 拟人化（间隔 ≥ 5 秒、每号每日上限、不 24h 跑）；失败即冷却数天。
- 平台分级：耐造档（B站 / 知乎 / 贴吧 / 小红书）多采；易封档（抖音 / 快手 / 微博）轻碰低频。
- 122 是机房 IP 更易被风控；顶不住再上住宅代理（按需付费，非默认）。

---

## 3. 禁止范围

- 不做"帮客户发布到客户自己的号"（三期走官方扫码授权 / 开放平台，本期不碰）。
- **不托管客户平台 cookie**（零客户凭证保管；托管 = 脱库即集体盗号，商用灭顶风险）。
- 不强化自己的 CDP/Pro 去硬刚平台搜索风控（高危搜索交给江湖 / TrendRadar）。
- 不集成 last30days 爬虫 / apify-ultimate-scraper；**agent-reach 仅作 AI 线海外内容的 ad-hoc 阅读/探查**（同 Firecrawl，不入自动管线、不作生产采集器，产出入库须转成带来源 URL/工具/采集状态的 `content_sample`）。
- 不动现有模块化前端结构（用户已定，盯着改的）；不改 MediaCrawlerPro 源码；不新建平行样本表。
- `trigger-collection` 自动闭环本期不实现（仍 Phase 2-2 暂缓；结合封号问题，做也要极克制）。

4. 江湖导出 → `import-batch` 的样本带 workspace，`/api/samples/hot30?workspace=` 能按工作台筛到（链路可验）。

---

## 5. 红线

- 客户永远看不到 cookie / 采集工具名词；客户面只有"成品 + 选题信号"。
- 自有采集号只做低危深挖，绝不让 Pro 自动猛搜。
- 样本仅用于自有内容生产研究；二创换结构换表达、不洗稿照搬（对齐母题资产 spec）。
- 商用 AI 生成内容的算法备案 / 内容审核合规另立 spec，不在本期。
