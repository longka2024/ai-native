# 内容教练闭环 Spec（越用越强的文案系统）

Date: 2026-06-16

## 背景 / 北极星

系统已上线,小妹(≈第一个真实客户)用它改小红书文案,"有点效果但没达预期"。
根因审计(2026-06-16,file:line 级):能力大多"只接了根线头"——
- `humanizer-zh` 去 AI 味只在初稿跑一次(`rewrite-engine.js:39,777`);
- `dbs-ai-check` / `dbs-xhs-title` 注册了但是**死标签**,从不被调用(`server.mjs:2951,2979`);
- **cheat-on-content(持续优化闭环)根本没接**,只被写成一句 prompt(`rewrite-engine.js:140`),15 个 cheat-* skill 全在 `E:\Codex` 没迁过来;
- 3 轮编辑质检闭环因两个 bug **空转**(`rewrite-engine.js:809` 传参形状错、`810` 查 `result.pass` 但函数返回 `passed`);
- 内容策略/编辑诊断**硬编码死在美业**(`server.mjs:4295`、`content-creation-base.js:325`),其余 3 条线质检是瞎的。

**总原则(凌驾本 spec,见 `memory/` + Claude 记忆):个人工具箱 ≠ 商品。**
客户不懂技术、也不该懂;能力一律沉后端工作流,客户零技术只拿结果。
**粘性 = 数据复利护城河**:教练每发一篇更懂这个客户的号,越用越离不开。

**本质洞察(用户 + 外部佐证文章《你的公众号文章被判低创作…》):**
"别人没有的东西"= AI 唯一造不出来的东西(编=假,违反失败策略红线)。
增量素材只能来自客户本人;系统的任务是**把它掏出来、存起来、每次写作强制用上**,而非生成。

## 1. 名字

内容教练闭环（越用越强的文案系统）

## 2. 触发词

- 「教练闭环」「文案教练」「点评闭环」
- 「越用越强」
- 「独家素材库」「增量素材」
- 「rubric 进化」「评分越用越准」

## 3. 范围（可执行动作）

> 全部"沉后端、客户零技术、UI 用 Longka 大白话"。分期推进。

### P0 快赢（先解阻碍,让小妹立刻感受到提升）
1. 修编辑质检闭环两个 bug(`rewrite-engine.js:809-810`),让 3 轮质检真正在草稿上跑。
2. 复活 AI 味检测回炉:不达标 → 自动触发 humanizer 重写 → 复检。
3. `humanizer-zh` 去 AI 味**扩到每一版**(现仅初稿),且失败要可见、不静默吞。

### P1 教练点评 + 独家素材库（核心）
4. **教练点评(打分)**:后端按"**该客户的 rubric**"对草稿多维打分;客户侧用大白话展示"哪强哪弱 + 具体改哪几句"。底层借 cheat-on-content 的 `cheat-score` 方法论。
5. **独家素材库(增量层)**:每客户专属的经历/失败/独家数据/客户原话库。
   - 采集走 **A 随写随问**(写作流程里顺口问"这观点你踩过坑吗?花了多少时间/钱?客户常问哪句?",答了就存)+ **C 从存量扒**(导入过去的帖子/朋友圈/客户回复,自动提炼独家片段);**不走 B 问卷**(违背"方便好用")。
   - 写作时**强制嵌 ≥1 条客户真实独家片段**(扩展现有"逐字锚点"机制 `server.mjs:4498`,锚客户本人的料)。
6. **"增量/存在感"评分维度**:一篇全是网上能搜到的通用内容 → 判低分 + 提示"这篇没有你自己的东西,先别发"。
7. **话题"已用过"管理 + 复用必换角度**(2026-06-16 用户新增):
   - 小妹选过/写过的话题**自动标记"已用过"**,按**客户/工作台隔离持久化**(需先确认话题唯一标识与落点:PG `longka_content_samples` 关联 / `command-center.json` state)。
   - 话题列表**默认过滤掉已用话题**,UI 显示"已用过"角标。
   - 若她坚持复用同一话题:生成前**必须先分析上一版用过的切入角度**,强制从**另一个角度**给建议、再写一篇(母题复用 / 同题不同切入,直接复用第 6 条"增量/存在感"维度与 `memory/rubric-dimensions.md`)。

### P2 发布后复盘 + rubric 进化（数据复利）
7. 客户发布后填/系统抓真实数据 → 复盘(`cheat-retro`)→ **进化该客户的 rubric**(`cheat-bump`)。
8. **每客户一套独立、会进化的 rubric**(多租户隔离)。

### P3 朱雀 AI 味终检（增强,依赖可行性验证）
9. 两级 AI 味检测:本地正则粗筛(免费快)→ 腾讯朱雀终检(权威把关)。**前置:先验证朱雀有无可调接口**(主要是网页工具、需登录、疑无公开 API);可行才上线,不可行则本地 + `dbs-ai-check` 兜底并记录原因。

### 贯穿
10. UI 全程 Longka 大白话,**绝不露** skill / rubric / 盲预测 / cheat / 朱雀 等黑话(承接 `memory/longka_product_decisions.md`)。
11. 把 cheat-on-content 从 `E:\Codex\ai-native\external/` 迁入本仓库(沿用迁移 spec 的红线:gitignore、不进 git secret)。

## 4. 禁止范围

- ❌ 不让客户懂技术 / 不暴露 skill 名 / 不要求客户跑命令
- ❌ **不让 AI 编造客户的经历/数据/案例**(增量素材只能来自客户本人;编=假数据红线)
- ❌ 不做"改几个词换句子"式伪原创(文章佐证:原创度 ≠ 增量价值)
- ❌ 不动已认可的前端模块拆分结构;不改 MediaCrawlerPro 源码;不新建平行样本表
- ❌ rubric 升级**不许跳过**全量重打 + 跨模型审核(防自欺,cheat-on-content 三原则);盲预测发布前写死、不可改
- ❌ 多租户**不混用**客户 rubric(每客户独立)
- ❌ 朱雀接口可行性未验证前,不承诺集成方式、不硬刚反爬
- ❌ 本 spec 是产品方向 spec,各 P 阶段开发前仍各自复述边界后再写码

## 5. 验收标准

- **客户视角(零技术)**:写文案 → 看到大白话点评 + 改进建议 → 改 →(发布后)填数据 → 下次点评明显更准,全程不接触任何技术名词。
- **后端**:每客户一套独立 rubric 文件;发布复盘后能看到 rubric 真的变了(版本化可追溯)。
- 草稿**强制含 ≥1 条客户独家素材**;纯通用内容被判低分并提示"先别发"。
- 盲预测不可改;rubric 升级走全量重打 + 审核(否则拒绝升级)。
- P0(部分已发 2026-06-16):✅ humanizer 去 AI 味对每一版生效(含"继续优化"版)、✅ 编辑审查记录真实验收结论且不覆盖 LLM 好稿;⬜ 待做:AI 味不达标自动回炉重写。
- 话题"已用过":小妹选过的话题再进列表显示"已用过"角标且默认过滤;坚持复用时,生成的新稿与上一版**切入角度不同**(可对比角度差异)。
- P3:朱雀接口可行性有明确结论(可行则两级检测上线;不可行则兜底方案 + 记录原因)。

## 6. 执行优先级

P0 快赢(修 bug + 回炉 + humanizer 扩版)→ P1 教练点评 + 独家素材库(核心)→ P2 复盘 + rubric 进化 → P3 朱雀终检。
每阶段独立可验收;P0 可先单独排上让小妹尽快见效。

## 7. 设计参考(外部产品 Helio,2026-06-16)

用户分享的 Helio（AI Native Workforce）印证了本方向。可借鉴的设计点:
- **Activity / 全程可追溯**:agent 工作流每一步(发什么消息、调什么工具、到哪步、报错)都可追踪、可回到具体步骤修正——作为本系统透明度原则(呼应 `memory/longka_failure_policies.md` 的"不藏长任务/给证据/不造假")。
- **每客户独立、可搜可改可删的记忆**:印证 per-customer rubric 设计;"可删"对应 cheat-on-content"rubric 是工作台不是博物馆"。
- **agent 分工分解**(热点/选题/适配度/文案/金句/风格/分镜)可作为内容 agent 工作流的分步参考。
- **审批卡(approval workflow)**:高风险动作(对外发布、发邮件、代发消息)先路由给人,一键通过/编辑——贴合"客户只定方向、做把关"的定位,也是将来做发布时的安全闸。

**关键差异(我们的护城河,不可丢):** Helio 面向"会管 agent 的一人公司/自媒体人"(半技术、操作者级),把 agent/频道暴露给用户自己配置。**本产品的终端客户是不懂技术的中小商户**,目标是连 agent 都不暴露、点一下拿结果——Helio ≈ 我们"操作者层",我们要在其上再包一层零技术产品。**别照搬"让客户自己管 agent"的 UX(那会退回个人工具)。** 叠加我们的**数据复利护城河(每客户 rubric 越用越准)**,定位更下沉、更产品化。
