# Longka AI Native 开发协作手册

## 1. 核心原则

Longka AI Native 的开发方式必须从“AI 自由发挥”切换为“标准先行、验证闭环”。

以后任何功能开发都必须先回答：

- 这个功能服务哪个用户动作
- 输入是什么
- 输出是什么
- 数据存在哪里
- 页面上用户怎么操作
- 成功标准是什么
- 如何验证不是假结果

没有对齐这些问题，不进入编码。

## 2. 必须使用的工作流

每个功能默认按以下顺序推进：

```text
需求澄清
-> 标准对齐
-> 最小可用设计
-> 编码
-> 本地验证
-> 122 验证
-> 记录经验
-> 提交仓库
```

不能跳过标准对齐直接写代码。

## 3. Superpowers 的使用规则

Superpowers 不是装饰品，是开发约束。

对应关系：

- brainstorming：用户需求模糊时，先把目标、边界、输入输出说清楚。
- writing-plans：复杂功能拆成短任务，避免一次性写一大坨。
- tdd：高风险逻辑先写可验证用例，再改实现。
- systematic-debugging：遇到 bug 先复现和定位，不凭感觉补丁。
- verification-before-completion：没有验证，不说完成。

项目专属规则只写 Longka 特有约束，不重复 superpowers 已经规定的通用流程。

## 4. Harness 的使用规则

Longka harness 的作用是把项目经验变成硬规则。

必须长期保留的规则：

- 不允许假数据冒充真实采集结果。
- 不允许只在本机验证就说团队可用。
- 不允许在今日工作台外随意新开割裂页面。
- 不允许用内部术语污染客户界面。
- 不允许把内容生成做成固定模板。
- 不允许文案优化只追加建议。
- 不允许采集模块只服务某个临时按钮。
- 不允许乱码文件长期留在项目中。

这些规则要进入开发检查清单。

## 5. 122 验收环境

122 是团队测试和演示环境。

当前约定：

- 公网地址：`http://122.51.218.154/ai-native-v2/workbench-v2.html`
- 服务名：`ai-native-command-center-v2`
- 服务路径：`/home/ubuntu/ai-native-command-center-v2`
- 本机远程执行工具：`python E:\Codex\remote_exec.py 122 "<command>"`

功能只有在 122 上验证通过，才可以说团队可测。

本地 `127.0.0.1` 只能算开发验证，不能算交付验证。

## 6. 数据库原则

122 使用 PostgreSQL，作为团队版内容资产库和未来训练语料库。

本地客户包未来使用 SQLite，作为轻量分发版本。

两者职责不同：

- PostgreSQL：大样本、团队协作、训练数据、长期内容资产。
- SQLite：单客户、本地运行、小规模资产、便携分发。

开发时必须明确当前功能写入哪一类数据库，不能让数据到处飞。

## 7. 内容资产库开发标准

内容资产库是全局基础设施，不是某个页面的临时缓存。

任何采集结果都应该进入统一结构：

- source_samples：原始内容样本
- source_metrics：互动数据
- source_comments：评论与用户问题
- content_labels：标签和人工判断
- topic_candidates：母题候选
- title_assets：标题资产
- content_fingerprints：平台指纹
- rewrite_versions：改写版本
- publish_records：发布记录

页面读取素材时，必须说明数据来源：

- 新采集批次
- 历史资产
- 手工导入
- 收藏导入

不能让用户误以为历史资产是刚采集的新数据。

## 8. 采集工程开发标准

采集工程要为全局服务。

每个平台都要定义：

- 用什么工具采集
- 采集什么对象
- 优质样本标准
- 过滤规则
- 详情抓取方式
- 评论抓取方式
- 失败重试方式
- 入库字段
- 风控和频率控制

示例：

- 小红书：优先 MediaCrawlerPro，重点抓标题、正文、图片、评论、互动数据。
- X：优先 XCrawl，重点抓指定账号优质帖、长帖、互动高帖、可收藏帖。
- 公众号：后续结合导入、RSS、商业采集或手工标注。
- RSS / Newsletter：作为持续信息源监控。
- Obsidian / X 书签：作为历史精选资产导入。

采集页面必须显示过程，不能让用户干等。

## 9. 内容生产开发标准

内容生产不能退化成固定模板。

必须接入内容创作基座：

- dbs 内容诊断
- dbs 标题公式
- Cheat on Content 评分思路
- Humanizer 去 AI 味
- 用户问题库
- 爆文拆解库
- 标题资产库
- 平台适配规则

生成前必须先判断：

- 这篇内容的主问题是什么
- 用户为什么关心
- 适合哪种内容结构
- 标题应走哪个角度
- 是否有足够素材支撑

生成后必须检查：

- 标题正文是否一致
- 是否有 AI 味
- 是否像平台内容
- 是否有信息增量
- 是否有真实语境
- 是否有明确行动入口

## 10. UI 开发标准

页面设计必须从用户动作出发。

禁止：

- 只画流程图，不驱动用户行动。
- 用大量卡片堆信息。
- 内部术语直接展示给客户。
- 把功能拆到割裂页面让用户迷路。
- 按钮没有反馈。
- 长时间等待没有进度提示。
- 下一步动作不明确。

每个页面都必须回答：

- 我现在在哪一步
- 我为什么要做这一步
- 我该填什么
- 点完会发生什么
- 出错怎么办
- 下一步去哪

## 11. 编码规范

编码必须使用 UTF-8。

不要继续制造乱码文件。

新增文件前先判断是否必须新增。能在现有清晰模块中完成，就不要再创建割裂文件。

清理原则：

- 删除已经不被引用的临时文件。
- 删除旧兜底假数据。
- 删除已废弃的测试入口。
- 保留能跑通系统的最新代码。
- 不动与当前任务无关的用户改动。

## 12. Bug 修复规范

遇到 bug 时按这个顺序：

1. 复现问题。
2. 判断是前端、后端、数据库、采集、部署还是缓存。
3. 找到真实入口和真实数据流。
4. 最小范围修改。
5. 本地验证。
6. 122 验证。
7. 记录原因和防复发规则。

禁止一边猜一边到处补。

## 13. 提交规范

提交前必须检查：

- 当前改动是否与任务相关
- 是否包含敏感信息
- 是否包含假数据
- 是否新增乱码
- 是否在 122 验证
- 是否更新必要文档

提交信息要说明用户价值，不只写“fix bug”。

## 14. 完成标准

一个开发任务完成，必须同时满足：

- 用户目标被满足
- 页面流程能跑通
- 数据真实
- 本地验证通过
- 122 验证通过
- 必要文档更新
- 没有明显冗余代码
- 用户能看懂下一步

如果没有做到，只能说“已完成某一部分”，不能说“做好了”。
