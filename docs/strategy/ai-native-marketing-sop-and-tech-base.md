# Longka AI Native 营销中台标准 SOP 与技术基座

更新时间：2026-05-29

## 0. 总原则

AI Native 中台不是页面上摆几个 Agent，也不是堆一批爬虫、视频、文案工具。

它必须跑通一条业务闭环：

```text
真实数据输入
-> 机会识别
-> 内容/视频生产
-> 发布包交付
-> 人工确认发布
-> 数据回填
-> 复盘优化
```

每一步都必须有输入、工具、产出物和验收标准。没有真实数据、没有可交付结果、没有复盘，就不算跑通。

## 1. 数据采集层：爆款侦察兵

### 目标

先找到市场已经验证过的流量信号，而不是凭感觉写内容。

### 输入

- 行业关键词。
- 竞品账号。
- 头部账号。
- 黑马账号。
- 平台搜索结果。
- 评论区。
- 私域反馈。
- 自己历史内容数据。

以色彩项目为例，第一批关键词：

- 形象分析
- 色彩分析
- 个人风格测试
- 发型建议
- 穿搭风格
- 拍照风格
- AI 形象照
- 素人改造

### 技术基座

- `agent-reach`：小红书、抖音、微博、B 站、Reddit 等平台轻量采集。
- `xhs-cli`：小红书搜索、笔记详情、评论读取。
- `xcrawl MCP`：复杂网页、动态页面、搜索结果页抓取。
- `MediCrawler`：平台内容和评论采集候选基座。
- `Apify`：复杂站点和海外平台采集候选基座。
- `wechat-radar / wechat-assistant`：私域聊天、群聊需求、客户反馈。

### 产出物

每条样本至少结构化为：

- 平台
- 关键词
- 标题
- 正文
- 标签
- 作者
- 发布时间
- 点赞数
- 收藏数
- 评论数
- 评论区高频问题
- URL / ID
- 截图或封面

### 验收标准

第一阶段不追求全自动，先追求真实可用：

- 每个关键词至少拿到 10 条真实样本。
- 每天至少沉淀 30 条样本。
- 至少筛出 5 条高价值爆款样本。
- 高价值样本必须有标题、正文、互动数据、评论区信息。
- 不能只有链接，必须转成可分析的结构化 JSON/Markdown。

### 当前缺口

目前中台页面里的“爆款侦察”主要还是静态示例，没有真实调用采集工具。这是第一优先级缺口。

## 2. 拆解分析层：爆款拆解员

### 目标

把采集来的样本拆成可复用的内容规律，而不是简单复制。

### 输入

- 爆款样本库。
- 评论区痛点。
- 客户业务资料。
- 产品卖点。
- 目标人群。

### 技术基座

- Codex / Claude：结构化分析和规则抽取。
- `longka-content-intelligence`：内容情报与爆款拆解 SOP。
- `social-content`：社媒内容反向工程。
- `read / learn / write`：长文、网页、案例拆解。
- `Waza`：把拆解出的规律沉淀成可迭代 SOP。

### 产出物

每条高价值样本输出：

- 爆款标题公式。
- 核心选题角度。
- 内容框架和逻辑。
- 黄金 3 秒开头。
- 高互动结尾引导。
- 评论区用户痛点。
- 可迁移到我们产品的角度。
- 不适合照搬的风险点。

### 验收标准

- 不能只说“这条不错”。
- 每条样本必须拆出明确结构。
- 至少形成 5 条可复用标题公式。
- 至少形成 5 类用户痛点。
- 至少形成 3 种内容结构模板。
- 必须能直接进入下一步内容生成。

## 3. 机会判断层：选题调度员

### 目标

从一堆热点里判断今天真正该做什么。

### 输入

- 爆款拆解结果。
- 当前产品状态。
- 当前素材库。
- 当前转化目标。
- 老板今天想推的业务。

### 技术基座

- Codex / Claude：机会评分、任务拆解。
- Hermes / agency-orchestrator / multica：后续可做任务调度。
- `Waza`：复盘后优化判断规则。
- 本地业务知识库：老板判断标准、客户资料、历史发布结果。

### 产出物

- 今日可做选题列表。
- 每个选题的推荐平台。
- 推荐内容形态：图文 / 视频 / 朋友圈 / 公众号。
- 推荐理由。
- 需要的素材。
- 风险提示。
- 预估转化目标。

### 验收标准

- 每天不输出一堆泛泛选题，只输出 3-5 个可执行选题。
- 每个选题必须说明为什么今天做。
- 每个选题必须匹配现有素材。
- 每个选题必须能派发给文案员工或视频员工。

## 4. 文字生产层：内容策划员

### 目标

把选题变成可发布、可复制、可审核的文字内容。

### 输入

- 选题。
- 爆款公式。
- 客户业务资料。
- 平台要求。
- 老板表达风格。

### 技术基座

- Codex / Claude：主生成。
- `write` / `humanizer-zh`：中文润色、去 AI 味。
- `social-content`：平台内容格式。
- `md2wechat-skill`：公众号排版。
- `Kami`：报告、发布包、卡片化交付。
- `GEOFlow`：搜索可见性和结构优化。

### 产出物

每个选题至少输出：

- 3 个标题。
- 1 篇小红书笔记。
- 1 条朋友圈配文。
- 1 条短视频口播稿。
- 1 个封面标题。
- 1 组标签。
- 1 条评论区引导话术。

### 验收标准

- 不能只有草稿，要能直接复制发布。
- 小红书正文不能空泛，必须包含场景、痛点、解决过程、行动引导。
- 朋友圈不能像广告，要像真实表达。
- 口播稿必须适合 30-60 秒视频。
- 每条内容必须标注适用平台和使用方式。

## 5. 视觉与图片生产层：图像设计员

### 目标

把文字内容变成有点击欲望的封面、图卡、报告图和分享图。

### 输入

- 文案。
- 标题。
- 产品截图。
- 客户案例图。
- 报告图。
- 风格参考。

### 技术基座

- GPT Image / gpt-image-2 风格库。
- Open Design / html-anything：网页和视觉表达。
- baoyu-xhs-images：小红书图卡生成。
- high-end-visual-design / frontend-design / taste-skill：审美和页面质量控制。
- personal-color-stylist：色彩项目报告和形象图经验。

### 产出物

- 小红书封面。
- 朋友圈分享图。
- 图文卡片。
- 报告样片图。
- 产品流程截图说明图。

### 验收标准

- 第一眼能看懂主题。
- 标题不折行、不拥挤。
- 图文不遮挡关键主体。
- 样式符合平台，不像 PPT 截图。
- 能和视频封面共用一套视觉语言。

## 6. 视频生产层：视频导演员

### 目标

把选题、文案、录屏、图片、案例变成真正能发的视频，而不是图片轮播。

### 输入

- 小程序录屏。
- 产品操作视频。
- 客户案例素材。
- 报告图。
- 文案脚本。
- 封面标题。
- 配音文本。

### 技术基座

- Remotion：稳定批量模板。
- HyperFrames：高级片头、标题卡、转场、CTA。
- video-use：自然语言导演式剪辑、录屏剪辑、质检、重剪。
- FFmpeg / ffprobe：合成、混音、转码、抽帧。
- MiniMax TTS：旁白。
- KIE / Suno：背景音乐。
- MoneyPrinterTurbo：批量变体、字幕、节奏、素材门禁参考。

### 产出物

- 30-60 秒小红书/视频号/抖音视频。
- 视频封面。
- 旁白音频。
- 背景音乐。
- 字幕。
- 素材使用清单。

### 验收标准

- 前 3 秒必须有明确钩子。
- 中间必须有证明点，不能只闪图。
- 录屏类视频必须看得懂操作路径。
- 单客户案例不能混入其他人照片，除非明确标注为案例库。
- 字幕不能遮挡人脸和报告重点。
- 结尾必须有 CTA。
- 成片前必须抽帧检查：1 秒、3 秒、中段、结尾。

## 7. 合规质检层：发布审核员

### 目标

降低违规、夸大、侵权、低质、重复内容风险。

### 输入

- 图文内容。
- 视频成片。
- 标题。
- 标签。
- 封面。
- 使用素材。

### 技术基座

- Codex / Claude：规则审核。
- 平台敏感词库。
- video-use：视频抽帧质检。
- ffprobe：视频参数检查。
- 小程序审核经验、微信支付经验、内容合规经验。

### 产出物

- 可发 / 修改后发 / 不建议发。
- 风险原因。
- 修改建议。
- 人工终审清单。

### 验收标准

- 高风险行业必须人工终审。
- 不能承诺确定效果。
- 不能使用未授权肖像。
- 不能夸大 AI 能力。
- 不能出现平台明显敏感表达。

## 8. 发布包交付层：运营交付员

### 目标

把内容整理成老板或小妹能直接使用的发布包。

### 输入

- 审核通过的图文。
- 审核通过的视频。
- 封面。
- 标签。
- 发布时间建议。

### 技术基座

- Kami：发布包包装。
- md2wechat：公众号排版。
- html-anything：网页交付页。
- 本地工作台 / U 盘工作台。
- 小程序轻量交付页。

### 产出物

一个发布包至少包含：

- 平台。
- 标题。
- 正文。
- 标签。
- 封面。
- 视频或图片。
- 发布时间建议。
- 复制按钮。
- 风险提示。
- 发布后数据回填入口。

### 验收标准

- 小妹不需要理解底层工具。
- 内容复制到小红书/朋友圈不乱格式。
- 文件路径清晰。
- 视频、封面、文案放在同一个任务包里。
- 发布状态可记录。

## 9. 数据回填层：运营复盘员

### 目标

让每次发布都变成下一次优化的依据。

### 输入

- 发布时间。
- 平台链接。
- 曝光。
- 点赞。
- 收藏。
- 评论。
- 私信。
- 小程序搜索。
- 试看数。
- 付款数。
- 成交反馈。

### 技术基座

- 本地数据库 / JSON / CSV。
- 小程序订单数据。
- 后台统计。
- 手动回填表。
- Codex / Claude 复盘分析。

### 产出物

- 每日复盘。
- 爆款共性。
- 低效内容原因。
- 明日选题建议。
- 模板优化建议。
- 产品流程优化建议。

### 验收标准

- 每条发布内容必须有状态。
- 每日必须能看出哪条内容带来真实线索。
- 不能只看点赞，必须关联咨询、试看、付款。
- 复盘结果必须反哺选题和模板。

## 10. 当前最小闭环

不要一上来做全平台矩阵。

当前最小闭环是：

```text
小红书色彩项目爆款侦察
-> 拆 5 条高价值样本
-> 生成 3 条小红书笔记
-> 生成 2 条视频脚本
-> 用小程序录屏 + 报告样片做 1 条视频
-> 生成发布包
-> 小妹人工发布
-> 回填数据
-> 第二天复盘
```

这条闭环跑通，才算 AI Native 营销中台第一阶段成立。

## 11. 分阶段标准

### P0：假数据演示

页面能展示流程，但不能真实采集。

状态：只能用于讲概念，不能收费。

### P1：手动触发真实采集

输入关键词，手动触发采集，生成爆款拆解报告。

状态：可以内部使用，验证方向。

### P2：半自动内容生产

真实样本进入内容生成，输出发布包，人工发布。

状态：可以给小妹使用，可以作为 U 盘版雏形。

### P3：每日工作台

每天自动生成今日选题、内容草稿、视频任务、发布清单、复盘表。

状态：可以卖给小微老板试用。

### P4：中台化

多项目、多行业、多账号资料、多素材库、多 Agent 任务编排。

状态：进入正式 AI Native 经营系统。

## 12. 对外产品语言

不要卖：

- 爬虫
- Agent
- MCP
- Prompt
- 自动化脚本
- 技术基座

要卖：

- 每天帮你找选题。
- 每天帮你写内容。
- 每天帮你做视频。
- 每天帮你整理发布包。
- 每天帮你复盘哪条内容更容易带来客户。

对老板的一句话：

```text
你不用学 AI，也不用每天从零想内容。系统每天帮你从真实市场里找机会，生成能发的图文和视频，你只需要审核、发布、看结果。
```

## 13. 2026-05-29 采集适配器修订

看完 X 上“小红书爆款复刻 Agent”案例后，采集层必须修正：

不要把 `xhs-cli` 写死成唯一采集方案。正确设计是“采集适配器层”。

工具优先级：

1. `apify-xhs`：优先用于小红书爆款笔记、评论、互动数据采集。适合 P1/P2 的真实样本获取。
2. `xcrawl-web`：用于公开网页、动态页面、竞品页面、搜索结果页和可访问链接。
3. `manual-import`：当平台风控或登录限制导致抓取失败时，让用户粘贴截图、正文、评论、链接，继续完成分析。
4. `xhs-cli`：仅作为本地登录态下的低频验证工具，不作为主生产采集基座。

统一输出结构：

```json
{
  "platform": "xiaohongshu",
  "keyword": "色彩分析",
  "title": "",
  "content": "",
  "tags": [],
  "author": "",
  "publishedAt": "",
  "metrics": {
    "likes": 0,
    "collects": 0,
    "comments": 0,
    "shares": 0
  },
  "comments": [],
  "url": "",
  "cover": "",
  "sourceTool": "apify-xhs | xcrawl-web | xhs-cli | manual-import",
  "collectedAt": ""
}
```

失败处理也必须产品化：

- 抓不到不能装作抓到了。
- 必须显示失败原因，例如登录态缺失、平台风控、链接不可访问、评论需要授权。
- 必须给替代方案：上传截图、复制正文、复制评论、换公开链接。
- 手动导入的材料也要进入同一套爆款拆解流程。

因此，爆款侦察兵 V1 的真正 SOP 是：

```text
输入关键词或爆款链接
-> 优先尝试 Apify
-> 失败则尝试 xcrawl
-> 再失败则要求手动导入
-> 统一结构化
-> 拆解标题公式、内容结构、黄金 3 秒、评论痛点
-> 生成复刻方案、小红书正文、视频脚本、评论区运营策略
-> 保存为可复用 SOP
```
