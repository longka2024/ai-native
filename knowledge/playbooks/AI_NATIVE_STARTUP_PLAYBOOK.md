# AI Native 创业工作法

这份文档沉淀 AI Native 创业的核心方法：几个人借助 AI 做过去几十上百人才能做的事，但前提是不要把“能建”误认为“该建”。

参考来源：

- 本地剪藏：`F:\Longka Wiki\龙咖Wiki\Clippings\AI Native 创业手册.md`
- 原始来源：Anthropic Founder Playbook / `claude.com/blog/the-founders-playbook`

## 1. 核心判断

AI 时代创业，最稀缺的资源从：

```text
能不能建
```

变成：

```text
该不该建
```

AI 让执行成本大幅降低，所以创始人的核心竞争力变成判断力、取舍能力、验证能力和编排能力。

## 2. 四个阶段

AI Native 创业可以按四个阶段推进：

```text
想法 -> MVP -> 上线 -> 规模化
```

每个阶段都要有退出标准，不能因为 AI 让开发变快，就跳过验证。

## 3. 想法阶段

目标：

```text
找到问题-方案匹配
```

先回答：

- 这个问题是否真实、具体、频率够高？
- 谁有这个问题？
- 这是否是一个市场？
- 他们现在怎么解决？
- 现有方案哪里不够好？
- 我们的方案是否真的解决这个问题？

退出标准：

- 能清楚说出谁有问题。
- 能说出问题发生频率和严重度。
- 有真实对话或公开证据支持。
- 不是只靠自己的直觉。

关键原则：

> 在证据支撑之前，不要急着建。

## 4. MVP 阶段

目标：

```text
把验证过的问题转成真实用户会使用的最小产品
```

MVP 不是堆功能，而是验证解决方案本身。

要避免：

- Agentic 技术债
- 范围蔓延
- 被早期热闹误判为 PMF
- 安全和隐私基础不足

MVP 前必须有：

- 架构原则
- 范围定义
- 不做什么
- 衡量指标
- 安全检查

退出标准：

- 用户回来再用
- 用户愿意付费
- 用户愿意推荐
- 有真实 PMF 迹象

## 5. 上线阶段

目标：

```text
把产品推向真实市场，并建立持续反馈回路
```

上线不是结束，而是更强的验证。

需要：

- 明确发布渠道
- 明确目标用户
- 明确衡量指标
- 明确反馈收集方式
- 明确下一轮迭代规则

不要把一次流量、朋友圈支持、短期热度误判为真正需求。

## 6. 规模化阶段

目标：

```text
让可验证的模式重复发生
```

规模化不是单纯加人，而是：

- 自动化流程
- 标准化 SOP
- 复用代码和模块
- 复用内容和运营模板
- 复用销售与交付流程
- 建立可复制增长引擎

## 7. 创始人角色变化

过去：

```text
创始人亲自干活
```

现在：

```text
创始人指挥 Agent 和系统干活
```

创始人要做的是：

- 判断什么值得做
- 设计验证路径
- 编排 AI 工具
- 维护业务边界
- 定义质量标准
- 决定什么时候停止、转向或加码

## 8. 对 Longka Harness 的启发

我们的长期基建要服务这个目标：

```text
少数人 + AI + Harness = 小团队做大团队的事
```

所以每个新项目不能直接进入开发，要先通过：

1. 市场方向验证
2. 用户需求证据
3. 竞品与替代方案分析
4. MVP 范围定义
5. 架构原则
6. 衡量指标
7. 安全/合规检查

## 9. 新项目强制问题

任何新项目启动前，必须先回答：

1. 这个问题真实存在吗？
2. 谁有这个问题？
3. 他们现在怎么解决？
4. 为什么现有方案不够好？
5. 我们有什么独特优势？
6. 最小可验证产品是什么？
7. 我们刻意不做什么？
8. 第一个用户如何判断有价值？
9. 付费或转化证据是什么？
10. 如果三轮迭代没有信号，如何停止或转向？

## 10. AI Native 的风险

AI 让开发太快，所以更容易犯这些错：

- 建没人要的东西
- 过早扩功能
- 用漂亮原型骗过自己
- 用短期热度误判 PMF
- 没有架构约束导致技术债
- 没有安全检查就上线
- 没有指标就不知道好坏

解决办法：

- 先验证，再建设。
- 先定义范围，再开发。
- 先定义指标，再上线。
- 先建立反馈，再规模化。

## 11. Longka 的执行原则

以后做项目时，默认流程是：

```text
证据 -> 范围 -> 架构 -> MVP -> 上线 -> 反馈 -> 规模化
```

Codex 和 skills 的作用不是让我们跳过判断，而是让我们更快拿到证据、更快试错、更快把有效路径产品化。
