# Skill 产品化与治理手册

这份文档用于把项目经验、业务 SOP 和工具脚本提纯成专业级 skills。目标不是写一堆提示词，而是沉淀可复用、可审计、可迁移、可售卖的 Agent 能力包。

参考来源：

- 本地剪藏：`F:\Longka Wiki\龙咖Wiki\Clippings\我写了半年skill，直到上周才意识到自己从一开始就搞错了方向.md`

## 1. 核心判断

Skill 不是“更长的提示词”。

Skill 是 Agent 运行时架构的一部分，是一个按需加载的专题工作流模块。

一个专业级 skill 应该包含：

- 明确的触发描述
- 标准流程
- 分支判断
- 失败处理
- 脚本入口
- 参考资料
- 示例输入输出
- 质量标准
- 安全边界

如果只是把一大段提示词放进 `SKILL.md`，那还不算专业 skill。

## 2. Agent 资产分层

不要把所有经验都塞进一个文件。要按层管理。

### 1. Memory 层

放长期偏好和跨项目经验。

适合放：

- 用户长期偏好
- 反复纠正过的问题
- 已验证成功的技术路线
- 跨项目通用经验

不适合放：

- 复杂流程
- 大段 SOP
- 项目专属长文档

### 2. AGENTS.md / CLAUDE.md 层

放每次会话都应该知道的项目地图和默认行为。

原则：

- 短
- 普适
- 高价值
- 低歧义

适合放：

- 项目结构
- 关键目录
- 默认开发流程
- 不可违反的基础约束
- 最常见坑点

不适合放：

- 长篇专题流程
- 某个任务才用到的 checklist
- 大段工具说明

### 3. Nested AGENTS.md / Path Rules 层

放只对某个目录或某类文件生效的规则。

适合放：

- 特定模块的代码风格
- legacy 目录规则
- admin 目录规则
- video 项目规则
- mobile 目录规则

不要把局部规则污染到全局。

### 4. Skill 层

放可复用的专题工作流。

适合放：

- 多步流程
- 分支判断
- 专题检查
- 专题知识
- 标准操作顺序
- 质量门
- 调用脚本的说明

例如：

- 小红书爆款拆解
- 评论区需求洞察
- 图文爆款生成
- 视频脚本复刻
- 微信小程序审核
- 宣传视频生成
- 个人色彩报告生产

### 5. Tools / CLI / MCP / Scripts 层

放真正执行动作的工具。

原则：

> Skill 负责判断什么时候用、按什么顺序用、结果怎么验收；脚本和工具负责真正执行。

不要在 skill 里堆一堆手写命令。应该把高频动作封装成 CLI、脚本、MCP 或 API。

### 6. Hooks 层

放必须强制执行的确定性约束。

适合放：

- 提交前必须跑 lint
- 数据库迁移前必须 dry-run
- 删除前必须检查路径
- 发布前必须检查环境变量

文档是建议，hook 是强制。

### 7. Subagents 层

放需要隔离上下文的任务。

适合：

- 大量读取代码
- 专项审查
- 并行验证
- 竞品资料整理
- 大规模文档归纳

### 8. Eval / Review 层

放反馈和评估。

作用：

- 发现 skill 是否真的触发
- 发现流程是否有效
- 发现输出是否达标
- 让 skill 持续迭代

## 3. 经验迁移规则

每次真实任务完成后，都要问：

> 这次学到的东西应该进哪一层？

判断规则：

### 进 AGENTS.md

如果它是每个会话都应该知道的基础规则。

例如：

- 这个项目用 pnpm
- 支付状态不能绕过数据库
- 生产图不能在未付款前触发
- 部署路径固定在哪里

### 进 Skill

如果它是多步流程、专题 checklist、分支决策。

例如：

- 小红书爆款拆解流程
- 视频爆款复刻流程
- 微信支付接入流程
- 后台订单排查流程

### 进 Nested AGENTS.md

如果它只对某个目录或模块生效。

例如：

- `promo-video/` 里的 Remotion 规则
- `server/` 里的订单状态规则
- `admin/` 里的后台操作规则

### 进 CLI / Script

如果它需要真实执行、查询、批量处理。

例如：

- 抓小红书 JSON
- 生成日报
- 批量压缩图片
- 检查订单异常

### 进 Hook

如果它必须每次强制执行，不能靠 Agent 自觉。

例如：

- commit 前跑测试
- deploy 前确认目标路径
- 删除前校验目录边界

### 进 Memory

如果它是用户长期偏好或跨项目经验。

例如：

- 用户喜欢一行命令
- 用户不喜欢英文文档
- 用户重视把经验沉淀成资产

## 4. 专业级 Skill 标准

一个可产品化 skill 至少要有：

### 1. 定位

写清楚：

- 解决什么问题
- 不解决什么问题
- 适合谁
- 什么时候不要用

### 2. 触发描述

`description` 要写得非常清楚。

因为 Agent 默认只看到 name 和 description。description 写不好，skill 就不会稳定触发。

### 3. 标准流程

不能只有原则，要有步骤：

1. 输入是什么
2. 先检查什么
3. 怎么处理
4. 怎么分支
5. 怎么验收
6. 失败怎么办

### 4. 工具入口

高频动作要封装成：

- `scripts/`
- CLI
- MCP
- API
- 模板文件

### 5. 示例

至少包括：

- 示例输入
- 示例输出
- 好结果
- 坏结果

### 6. 质量门

定义什么叫合格：

- 数据够不够
- 文案有没有 AI 腔
- 图片是否可用
- 视频节奏是否达标
- 报告是否能交付

### 7. 安全边界

必须写清：

- 不读取密钥
- 不上传敏感代码
- 不绕平台规则
- 不批量骚扰
- 不删除用户文件
- 不越权操作

### 8. 可迁移性

换机器后能装、能跑、能验证。

要有：

- 安装说明
- 依赖说明
- 配置说明
- 验证命令
- 常见问题

### 9. 版本管理

可售卖 skill 必须有：

- 版本号
- changelog
- 测试案例
- 兼容说明

## 5. 内部 Skill 和可售卖 Skill

### 内部生产型 Skill

目标：

- 提升自己团队效率
- 支撑项目交付
- 快速迭代

可以接受：

- 文档稍粗
- 强依赖本地路径
- 和业务强绑定

### 外部售卖型 Skill

目标：

- 给别人安装使用
- 降低交付解释成本
- 形成商品

必须做到：

- 路径可配置
- 文档清楚
- 示例完整
- 安全边界清楚
- 不依赖私有环境
- 有安装和验证流程
- 有演示案例

## 6. Skill 产品化路线

项目经验转产品的路线：

```text
真实项目经验
-> SOP
-> 内部 skill
-> 脚本/工具封装
-> 真实案例验证
-> 文档和演示
-> 外部可售卖 skill
```

不要一开始就为售卖写大而全的 skill。

先在自己的项目里跑通，证明有用，再提纯。

## 7. 可优先产品化的 Skill

结合当前基建，优先级较高的有：

- 小红书爆款拆解 Skill
- 评论区需求洞察 Skill
- 图文爆款生成 Skill
- 视频爆款复刻 Skill
- 内容工业化 SOP Skill
- 个人色彩报告生产 Skill
- 宣传视频生成 Skill
- 微信小程序审核 Skill
- 支付接入检查 Skill
- 订单状态排查 Skill

## 8. Skill 安全治理

Skill 是供应链入口。

安装别人写的 skill，等价于允许别人把流程和脚本放进你的 Agent 环境。

治理要求：

- 不随便安装未知来源 skill
- 安装前看 `SKILL.md`
- 安装前看 `scripts/`
- 检查是否读取 `.env`
- 检查是否上传文件
- 检查是否执行危险命令
- 检查是否隐藏网络请求
- 尽量固定版本
- 对外售卖 skill 不得包含私钥、token、内部路径

## 9. 我们自己的 Skill 设计原则

以后新增或改造 skill，要遵守：

1. 先有真实需求，再做 skill。
2. 先跑通 SOP，再封装 skill。
3. Skill 负责流程，不负责硬编码业务秘密。
4. 工具动作尽量下沉到 scripts/CLI。
5. description 要能稳定触发。
6. 每个 skill 要有验收标准。
7. 每个 skill 要有安全边界。
8. 有商业潜力的 skill 要从一开始考虑可迁移性。

## 10. 每次任务后的治理问题

任务结束时问：

```text
这次经验该进哪一层？
```

选项：

- memory
- AGENTS.md
- nested AGENTS.md
- skill
- script/CLI
- hook
- eval/review
- project docs
- long-term harness

这个问题是长期基建持续变强的关键。
