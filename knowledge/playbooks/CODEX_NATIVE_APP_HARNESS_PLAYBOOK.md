# Codex Native App Harness 工作法

这份文档沉淀一个重要判断：Codex 之后，开发 AI 应用的逻辑变了。

过去我们常想“自己开发一个 Agent”。现在更合理的路径是：

> 不重新造 Agent，而是把 Codex 的判断能力、computer-use、浏览器操作、imagegen、生图、skills 等能力接进自己的业务系统。

## 1. 核心原则

把工作拆成两类：

### 需要判断的事，交给 Codex Agent

例如：

- 判断一个 TikTok 红人是否符合要求。
- 浏览首页后决定下载哪些视频。
- 判断一条内容是否值得复刻。
- 判断竞品页面有什么变化。
- 判断生成图质量是否合格。
- 调试应用、复现问题、修 bug。
- 看截图、看页面、看日志后决定下一步。

这类工作原本需要人看、人判断、人试错。现在适合交给 Codex/Computer Use/Chrome 类能力。

### 流程化的事，交给应用和脚本

例如：

- 批量下载视频。
- 批量抓取数据。
- 批量生成封面。
- 定时跑关键词监控。
- 排队生成图文。
- 多平台发布。
- 记录发布结果。
- 写入数据库。

这类工作不应该每次都让 Agent 慢慢操作，应该产品化成 App、脚本、队列和后台任务。

## 2. 新架构

推荐架构：

```text
业务 Harness
  -> Codex Agent 判断层
  -> Skills 能力层
  -> App / GUI 流程层
  -> 自动化任务层
  -> 数据反馈层
```

### Codex Agent 判断层

负责：

- 看页面
- 看截图
- 看日志
- 做选择
- 做诊断
- 做质量判断
- 做复杂调试

### Skills 能力层

负责把能力标准化：

- 小红书调研
- imagegen 生图
- Remotion 视频
- HyperFrames 布局
- Agent Reach 搜索
- 服务器部署
- 市场调研
- 爆款拆解

### App / GUI 流程层

负责让非技术人员也能用：

- 内容工作台
- 图文爆款机
- 视频爆款机
- 小红书调研台
- 竞品监控面板
- AI 画布
- 调试面板
- 任务进度可视化

### 自动化任务层

负责高频重复动作：

- 定时任务
- 队列
- 批量处理
- 自动报告
- 失败重试
- 成本控制

### 数据反馈层

负责复盘：

- 哪些选题有效
- 哪些图文有效
- 哪些视频有效
- 哪些标题失败
- 哪些内容带来私信/加微/付款

## 3. 为什么 App / GUI 很重要

纯 Codex 跑复杂流程很慢。

适合 Codex 的是判断和复杂任务，适合 App 的是高频流程。

所以要把多个 skills GUI 化，把它们变成小妹、运营、项目同事能点的工作台：

- 点按钮开始采集。
- 点按钮生成 3 个版本。
- 点按钮查看爆款拆解。
- 点按钮重做一张图。
- 点按钮发布到多个平台。
- 页面看到任务进度和失败原因。

这样才能从“能跑”变成“能量产”。

## 4. Codex 接入应用的意义

如果 App 能以 app-server 的方式接入 Codex 能力，就能做到：

- App 调 Codex 做判断。
- App 调 Codex 看浏览器/页面。
- App 调 Codex 生图。
- App 调 Codex 修 bug。
- App 把过程可视化。
- 人在手机端也能看进度、发命令、远程推进。

这会让应用随着 Codex 能力增强而增强。

关键原则：

> 业务逻辑属于我们，底层 Agent 能力可以借用最强的现成系统。

## 5. 对内容工业化系统的启发

内容系统可以这样设计：

```text
Agent 判断爆款机制
App 批量跑内容流程
Skills 负责具体能力
Harness 负责业务 SOP
数据库记录反馈
```

例如图文爆款机：

1. Agent 分析小红书爆款样本。
2. App 管理关键词、样本、任务队列。
3. imagegen 生成封面和案例图。
4. 文案 skill 生成标题和正文。
5. 发布模块生成多平台版本。
6. 反馈模块记录效果。
7. Harness 更新模板库。

例如视频爆款机：

1. Agent 分析对标视频节奏、镜头、音乐、文案。
2. App 管理素材、脚本、分镜和渲染任务。
3. Remotion / HyperFrames / FFmpeg 执行生产。
4. Agent 检查成片是否符合爆款机制。
5. 发布后记录数据，更新模板。

## 6. 对视觉生成能力的要求

图文爆款机、视频封面机、落地页视觉机都要接入高质量视觉生成能力。

当前应遵守 `imagegen` skill 的规则：

- 普通图片生成优先使用内置 imagegen。
- 项目资产必须保存到项目目录。
- 不把项目引用资产留在默认生成目录。
- 不随意降级模型。
- 批量图要按资产拆 prompt。
- 透明图先走 chroma-key 路线，必要时再确认 CLI fallback。
- 生成后必须检查主题、风格、文字、构图和路径。

未来可扩展：

- gpt-image2 高质量图片
- ComfyUI
- RunningHub
- Seedance
- Sora
- Remotion
- HyperFrames

工具可以换，但 SOP 和质量标准不能丢。

## 7. 对新项目的启发

以后做新项目，不要先问：

> 我要不要自己开发一个 Agent？

应该问：

1. 这个项目里哪些步骤需要判断？
2. 哪些步骤是流程化的？
3. 哪些能力已经有 skill？
4. 哪些流程需要 GUI 化？
5. 哪些任务要进队列？
6. 哪些数据要反馈到 SOP？
7. 如何让 Codex 能力接进业务系统？

这就是 Codex Native App Harness 的核心。

## 8. 长期目标

最终要形成：

```text
Longka AI Business Harness
```

它不是一个单一产品，而是一套生产系统：

- 用 Codex 做判断。
- 用 Skills 做能力复用。
- 用 App 做流程产品化。
- 用 Harness 沉淀业务 SOP。
- 用数据反馈持续进化。

以后 Codex 越强，接入这套 Harness 的应用就越强。
